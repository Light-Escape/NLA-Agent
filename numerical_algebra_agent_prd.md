# 数值代数 Agent 前端产品需求文档（PRD）

## 1. 产品背景

数值代数任务通常具有以下特点：

- 输入数据以文件形式存在，且文件类型多样（脚本、矩阵数据、实验日志、结果文件等）；
- 计算过程包含大量中间变量，需要随时查看变量状态；
- 结果输出往往包含数学公式、推导、伪代码和实验结论；
- 用户需要在“理解—修改—执行—验证”的循环中反复工作。

传统聊天式界面适合问答，但难以承载上述工作流。为提升数值代数 Agent 的实用性，需要构建一个更接近“交互式计算工作台”的前端界面，使其同时具备文档呈现、文件管理和变量监控能力。

## 2. 产品目标

### 2.1 总目标

构建一个面向数值代数任务的交互式 Agent 前端，支持 Markdown 和 LaTeX 渲染、文件列表展示与读取、以及类似 MATLAB Workspace 的变量信息面板，从而提升 Agent 的可读性、可操作性和可追踪性。

### 2.2 具体目标

1. 让 Agent 的输出可以以技术文档形式清晰呈现。
2. 让用户能够直接浏览、打开、定位和读取项目文件。
3. 让用户实时了解当前会话中的变量状态及基本数值特征。
4. 让 Agent 的计算过程具备回溯能力，便于调试、验证和复现实验。

## 3. 产品定位

该产品不是普通聊天窗口，而是一个“面向数值代数任务的交互式工作台”。

核心定位：

- **可读**：支持 Markdown、LaTeX、代码块、表格等富文本表达；
- **可查**：支持文件管理、快速预览、搜索和读取；
- **可追踪**：支持变量状态面板、中间结果记录和计算历史回放。

## 4. 用户画像

### 4.1 主要用户

- 数值线性代数研究者
- 科研助理或博士生
- 做算法原型验证的工程师
- 需要处理矩阵、稀疏结构、迭代算法和实验脚本的用户

### 4.2 典型使用场景

- 读取一组矩阵数据并运行分解算法；
- 观察迭代法的残差、步数和收敛趋势；
- 检查某个变量是否出现 NaN/Inf、维度错误或病态迹象；
- 对比不同参数下的实验结果；
- 生成适合写论文的公式化分析文本。

## 5. 需求范围

### 5.1 MVP 范围

第一版建议包含以下功能：

- Markdown 渲染；
- LaTeX 数学公式渲染；
- 文件列表展示；
- 文件内容预览与读取入口；
- 变量工作区面板；
- 变量基本信息展示；
- 对话内容与变量信息的联动跳转。

### 5.2 非 MVP 但建议规划的功能

- 变量版本历史；
- 变量分组；
- 文件全文搜索；
- 结果导出为报告；
- 变量之间的依赖关系图；
- 迭代过程曲线展示；
- 会话回放与实验快照。

## 6. 信息架构

建议采用三栏布局：

### 6.1 左侧：文件区

用于显示项目文件、目录树、最近文件、搜索结果。

### 6.2 中间：主内容区

用于显示 Agent 对话、Markdown 文档、公式、代码片段、结果摘要和解释。

### 6.3 右侧：Workspace 区

用于显示当前变量列表及其基本信息，并支持点击查看详细摘要。

## 7. 功能需求

### 7.1 Markdown 与 LaTeX 渲染

#### 功能描述

主内容区应支持渲染 Markdown 文本和数学公式，保证 Agent 输出具有论文式、报告式可读性。

#### 功能点

- 支持标题、列表、表格、引用、代码块；
- 支持行内公式与块级公式；
- 支持数学符号、分式、矩阵、上下标、极限、求和等常见表达；
- 支持长内容折叠与展开；
- 支持复制公式、复制代码块。

#### 验收标准

- 用户可正常阅读包含公式的回复；
- 常见 LaTeX 公式能够正确渲染；
- 公式与文本排版整齐，无明显错位。

### 7.2 文件列表与文件读取

#### 功能描述

左侧文件区展示当前项目中的文件和目录，支持快速打开、预览和读取。

#### 功能点

- 文件树浏览；
- 文件名搜索；
- 按类型过滤；
- 最近打开列表；
- 文件预览；
- 文件内容摘要；
- 支持将文件引用到对话上下文中。

#### 建议支持的文件类型

- `.m`, `.txt`, `.csv`, `.json`, `.mat`, `.npz`, `.py`, `.log`

#### 验收标准

- 用户能够快速找到目标文件；
- 文件内容可在界面内预览；
- Agent 能基于文件上下文进行回答。

### 7.3 Workspace 变量面板

#### 功能描述

右侧 Workspace 面板显示当前会话中的变量基本信息，帮助用户实时掌握计算状态。

#### 每个变量建议显示的信息

- 变量名；
- 类型（标量、向量、矩阵、稀疏矩阵、结构体等）；
- 尺寸；
- 数据类型；
- 是否稀疏；
- 复杂数/实数属性；
- 基本统计量；
- 最近更新时间；
- 来源（输入文件、用户编辑、算法生成等）。

#### 数值代数特有信息（建议）

- 范数；
- 非零元比例；
- 是否对称/厄米；
- 是否存在 NaN/Inf；
- 条件数估计；
- 残差历史；
- 迭代步数；
- 收敛状态。

#### 验收标准

- 用户能在不打开细节的情况下快速了解变量状态；
- 变量信息随计算过程实时更新；
- 明显异常能够被标记出来。

### 7.4 变量与结果联动

#### 功能描述

主内容区中的变量名或结果引用可点击，点击后自动定位到 Workspace 中对应变量。

#### 功能点

- 结果文本中的变量高亮；
- 点击变量跳转到变量详情；
- 变量详情中显示生成来源和相关计算步骤；
- 支持从变量反向定位到相关输出记录。

#### 验收标准

- 用户能够从结论快速追溯到变量状态；
- 用户能够从变量快速找到对应的解释文本或计算记录。

## 8. 变量元数据设计建议

为了保证前端渲染和后续扩展能力，变量不应只保存“值”，还应保存结构化元数据。

建议每个变量至少包含：

- `name`
- `type`
- `shape`
- `dtype`
- `is_sparse`
- `value_preview`
- `norm`
- `cond_est`
- `status`
- `source`
- `timestamp`
- `tags`
- `notes`

建议根据变量类型扩展字段，例如：

- 矩阵：稀疏度、对称性、三角性、谱信息；
- 迭代过程变量：残差序列、收敛标记、迭代次数；
- 文件变量：路径、大小、格式、是否已读取。

## 9. 交互设计原则

### 9.1 低干扰

界面不应过度复杂，默认展示最关键的信息，详细信息通过点击展开。

### 9.2 高可读

数学表达、变量摘要和文件信息需要保持一致的视觉层级，避免信息拥挤。

### 9.3 高可追踪

任何重要结果都应能够追溯到对应输入文件、变量和计算步骤。

### 9.4 面向科研工作流

界面设计应服务于“查看输入—运行计算—检查变量—验证结果—导出结论”的循环。

## 10. 非功能需求

### 10.1 性能

- 变量列表应支持大量变量的快速渲染；
- 文件树在文件数量较多时仍需保持流畅；
- 公式渲染不应明显阻塞主界面。

### 10.2 可扩展性

- 后续可以加入图表、依赖图、结果比较和报告导出；
- 元数据结构应支持新增字段而不破坏已有前端。

### 10.3 稳定性

- 变量信息更新失败时不应影响主对话区；
- 文件读取失败时应给出明确错误提示；
- 公式渲染异常时应有降级显示方案。

## 11. 风险与挑战

### 11.1 风险

- 变量信息如果过少，Workspace 价值有限；
- 变量信息如果过多，界面会变得拥挤；
- 文件与变量之间缺乏联动时，工作台感会很弱；
- 数值算法输出复杂，元数据标准化难度较高。

### 11.2 应对策略

- 默认显示摘要，详细信息按需展开；
- 对变量进行分类和标签化；
- 将文件、变量、输出记录统一到同一套元数据体系；
- 对数值代数常见对象设计专门的摘要模板。

## 12. 建议的版本规划

### V1

- Markdown/LaTeX 渲染；
- 文件树与文件预览；
- Workspace 变量列表；
- 变量基本信息展示；
- 变量与输出联动。

### V2

- 变量分组；
- 变量历史版本；
- 残差曲线和统计图；
- 文件全文搜索；
- 快速导出实验报告。

### V3

- 会话回放；
- 自动诊断建议；
- 变量依赖图；
- 多实验对比；
- 面向论文写作的结果整理。

## 13. 成功标准

该产品可被认为成功，如果满足以下指标：

- 用户能够在一个界面内完成“看文件—看变量—看结果”的主要工作流；
- 用户对 Agent 输出的信任度提高；
- 用户在调试数值实验时更少依赖外部工具；
- 变量和结果的追踪效率显著提升。

## 14. 结语

该前端的核心价值不在于“更好看”，而在于把数值代数 Agent 从聊天工具升级为可工作的计算环境。只要把 Markdown/LaTeX、文件管理和 Workspace 这三部分打通，这个产品就会具有非常明确的科研和工程实用性。
