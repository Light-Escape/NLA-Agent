# 工作区失配与工具幻觉问题分析

## 1. 问题是什么

数值代数 Agent 在处理大矩阵时，应该把矩阵放在工作区里，只通过句柄和摘要访问，而不是把完整矩阵搬进对话上下文。但实际运行中，Agent 会不断尝试绕过工作区，去获取完整矩阵、重建数据或寻找未声明的访问路径。这说明它没有稳定遵守工作区边界，也出现了工具幻觉。

## 2. 主要表现

- 反复想把矩阵完整读出来，再传给 Python 或重新重建；
- 在工具明确不可用后，仍继续猜测临时文件、共享内存、base64、逐列提取等绕路方案；
- 思考内容被“如何拿到数据”占满，而不是“如何在工作区内完成计算”；
- 多次失败后没有停下来，而是继续发散出新的访问方式。

## 3. 为什么这是问题

这会导致上下文被大矩阵污染、工具调用变多、推理偏离任务目标，也会让系统看起来不稳定。更重要的是，Agent 把不存在的访问路径当成可行方案，说明它没有真正理解工作区的权限边界。

## 4. 根因

核心原因有三个：工作区边界不够硬、失败后没有回退机制、没有固定的任务流程。模型一旦遇到矩阵任务，就容易默认“先把数据弄出来再说”，而不是先看摘要、再决定是否需要局部访问、最后在工作区内完成计算。

## 5. 怎么改

- 明确规定：只能使用工作区显式提供的接口，不能假设存在隐式通道；
- 失败后立刻停止当前路径，回到摘要和可用工具重新规划；
- 给常见任务做固定模板，优先走“摘要 → 决策 → 计算 → 汇报”的流程；
- 限制局部读取的规模和频率，避免通过多次小请求拼回完整矩阵。

## 6. 结论

这个问题本质上不是“模型不够聪明”，而是“系统没有把数据边界和工具边界约束严”。要解决它，重点不是让 Agent 更努力找数据，而是让它更严格地在工作区内做事。
